# AI 에이전트 작업 흐름 개선안

## 문제점: 2025-12-09 데이터 소실 분석 중 발생한 오류들

### 1. Project.gpr에 새 모듈 추가하지 않음
**발생한 일:**
- Test_DataPersistence.gpl 생성 후 Project.gpr에 등록 안 함
- Core_Main에서 호출했지만 컴파일 실패 (미정의 심볼)

**원인:**
- COMMON_MISTAKES.md 참고 안 함 (9.4 Project.gpr 등록 체크리스트)
- 모듈 생성과 프로젝트 등록을 분리해서 생각함

**개선:**
✅ 새 모듈을 생성할 때마다:
```
1. 파일 생성 (Test_DataPersistence.gpl)
2. Project.gpr에 ProjectSource 즉시 추가
3. 빌드 검증
```

---

### 2. 너무 빨리 추측하고 코드 수정함
**발생한 일:**
- "데이터가 사라진다" → 즉시 Merge/Sync 방식 변경
- 근본 원인 분석 없이 여러 번 수정
- XML 모듈(완성된)도 건드림

**원인:**
- 제대로 된 디버깅 프로세스 없음
- 테스트 파일 생성도 늦게 함

**개선:**
✅ 문제 분석 3단계:
```
1. 현재 상태 정확히 파악 (파일, 로그, 구조)
2. 근본 원인 찾기 (추측 금지)
3. 그 다음에 수정
```

---

### 3. 테스트 경로 vs 실제 데이터 경로 혼동
**발생한 일:**
- Demo.dat에서 Test123=123 저장
- 다음 실행 시 테스트 코드가 같은 파일 덮어씀
- 당신이 "데이터가 사라졌다"고 호소

**원인:**
- 초기 분석 부족 (테스트와 데이터를 분리해야 한다는 걸 못 봄)
- RunDatStoreTests() 코드를 제대로 읽지 않음

**개선:**
✅ 작업 시작 전에:
```
1. 기존 코드 완전히 읽기
2. 데이터 흐름 다이어그램 그리기
3. 그 다음에 수정 계획 세우기
```

---

### 4. XML과 DAT 불일치 방치
**발생한 일:**
- XML: Sync 방식 (Reset 있음)
- DAT: Merge 방식 (Reset 없음) 
- 일관성 없음

**원인:**
- 두 모듈을 동시에 검토하면서 불일치 발견했지만
- "XML은 완성됐다"고 해서 Merge로 강행함
- 나중에 Sync로 다시 변경

**개선:**
✅ 설계 일관성 우선:
```
1. 같은 역할의 모듈은 동일 패턴 사용
2. 불일치 발견 → 확인 후 통일
3. 완성 모듈도 필요하면 수정
```

---

### 5. 문서를 읽고도 적용 안 함
**발생한 일:**
- COMMON_MISTAKES.md와 ERROR_PREVENTION_CHECKLIST.md가 이미 있음
- 읽지 않고 내 추측대로 작업함

**원인:**
- 프로젝트 규칙을 무시하고 "빨리 처리"하려 함
- "전문가처럼"이라는 요청 후에야 문서 읽음

**개선:**
✅ 모든 작업 전에:
```
1. 프로젝트 구조 문서 읽기
2. 에러 방지 체크리스트 확인
3. 그 다음에 코드 수정
```

## 개선된 작업 흐름

### 문제 발생 시 (데이터 소실 예시)

**Step 1: 상황 파악 (rushing 금지)**
```
□ 문제 증상 정확히 이해
□ 관련 파일들 읽기 (core_Main, Data_DatStore, Data_XmlStore)
□ 데이터 흐름 추적
□ 로그/파일 내용 확인 요청
```

**Step 2: 근본 원인 분석**
```
□ 추측이 아닌 증거로 판단
□ 각 모듈의 역할 확인
□ 문서의 설계 의도 확인
□ 테스트 코드로 검증 계획
```

**Step 3: 수정 계획**
```
□ 일관성 검사 (XML ↔ DAT)
□ Project.gpr 확인
□ 수정 전 빌드 테스트 계획 수립
□ 당신과 협의
```

**Step 4: 구현 및 검증**
```
□ Project.gpr에 새 모듈 등록 (동시에!)
□ 수정 실행
□ 빌드 검증
□ 테스트 계획 실행
```

## 이번 세션에서 배운 점

| 실수 | 원인 | 개선 방법 |
|------|------|---------|
| Project.gpr 등록 지연 | 절차 누락 | 파일 생성 시 동시 등록 |
| 빠른 추측 | 성급함 | 데이터 수집 → 분석 → 수정 |
| 경로 혼동 | 코드 미읽 | 작업 전 전체 구조 파악 |
| 모듈 불일치 | 규칙 무시 | 일관성 우선, 문서 참고 |
| 규칙 미적용 | 문서 무시 | **모든 작업 전 문서 읽기** |

---

## 앞으로 적용할 우선순위

1. **프로젝트 규칙 우선** (문서 > 추측)
2. **데이터 수집 → 분석 → 수정** (역순 금지)
3. **한 번에 하나씩** (여러 수정 동시 진행 금지)
4. **검증 → 빌드 → 테스트** (순서 엄격)
5. **당신과 협의** (혼자 결정 금지)
